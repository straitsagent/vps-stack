# Portfolio Analysis Agent — Build Specification

*For Claude Code review and implementation on Contabo VPS (Ubuntu 24.04)*

*Spec version: 1.0 | Date: June 2026 | Source: Google Drive*

> **Status: Superseded.** This spec was the original design for a portfolio analysis agent. It has been superseded by the W-series Telegram agent (`/root/agent/`) + `research_tool` Windmill script. Kept for historical reference only — do not attempt to implement.

---

## 1. Project Overview

Build a weekly portfolio analysis agent as a Windmill workflow that:

1. Parses the latest portfolio email generated by the existing Portfolio Intelligence System
2. Fetches per-position research via web search API
3. Runs a multi-pass Claude-powered analysis agent over the full portfolio
4. Delivers a structured HTML analysis report via email

This workflow integrates into the existing Windmill automation stack and extends the Portfolio Intelligence System already running on this VPS.

## 3. Portfolio Email Structure

The existing portfolio emails are sent from straitsagent@gmail.com to <YOUR_RECIPIENT_EMAIL> with subject format:

```
Portfolio — {date} | {session} Close | ${total_value} | Day: {day_pnl}
```

### HTML Structure to Parse

The email body contains:

- **Header table:** Total Value, Day P&L
- **Top Movers section:** Up to 5 gainers and 5 losers with dollar P&L, percent change, portfolio impact
- **% Movers section:** Per-ticker news links from Google News RSS
- **All Positions table:** Full position table with columns: Ticker, Company, Shares, Price, Value (USD), Day P&L, Day %, Alloc %
- **Consolidated positions:** ALIBABA and BAIDU are multi-row with parent row (combined value) and child rows for HK listing and ADR (indented with ↳ prefix)

### Position Data Model

```python
@dataclass
class Position:
    ticker: str           # e.g. "GOOGL", "9988.HK", "ALIBABA" (consolidated)
    company: str
    shares: float
    price: float
    price_currency: str   # "USD" or "HKD"
    value_usd: float
    day_pnl_usd: float
    day_pct: float
    alloc_pct: float
    is_consolidated: bool        # True for ALIBABA, BAIDU parent rows
    sub_positions: list          # Child positions for consolidated rows
    exchange: str                # "NYSE", "NASDAQ", "HKEX"
```

## 5. Step-by-Step Implementation Spec

### Step 1: Fetch Portfolio Email

```
Fetch the most recent portfolio email from Gmail sent folder.

Use IMAP via u/admin/gmail_smtp (consistent with existing workflows 1.1 + Email Summary).
Alternatively, use Gmail API OAuth if a gmail_api resource is set up.

Search query: 'from:straitsagent@gmail.com subject:Portfolio in:sent'

Return the full HTML body of the most recent match.
Also return email metadata: subject, date, total_value, day_pnl
(parse from subject line).
```

**Inputs:** `$res:u/admin/gmail_smtp`

**Outputs:**
```json
{
  "subject": "str",
  "date": "str",
  "session": "str",
  "total_value_str": "str",
  "day_pnl_str": "str",
  "html_body": "str"
}
```

**Error handling:** If no email found in last 7 days, raise descriptive error. Do not silently fail.

### Step 2: Parse Portfolio Email

```
Parse the HTML email body into structured position data.
Use BeautifulSoup for HTML parsing.

Handle:
- Standard single-listing positions (most tickers)
- Consolidated positions (ALIBABA, BAIDU) with HK + ADR sub-rows
- HKD prices with USD equivalent in parentheses
- Negative and positive P&L values
- Percentage formatting (strip % and +/- signs)

Do NOT hardcode ticker names. Parse dynamically from table rows.
Identify consolidated rows by the presence of child rows with ↳ prefix.

Alternative approach (preferred): query portfolio_positions + price_history + fx_rates
directly from PostgreSQL and reuse P2 calculation logic. More robust, no HTML fragility.
```

**Inputs:** `html_body` from Step 1 (or `$res:u/admin/portfolio_db`)

**Outputs:**
```json
{
  "portfolio_summary": {
    "total_value_usd": 0.0,
    "day_pnl_usd": 0.0,
    "day_pnl_pct": 0.0,
    "position_count": 0,
    "date": "str",
    "session": "str"
  },
  "positions": [],
  "top_movers_up": [],
  "top_movers_down": []
}
```

**Validation:** Sum of position values_usd should reconcile to total_value_usd within 0.5% tolerance. Log warning if not.

### Step 3: Research Positions

```
For each position above RESEARCH_THRESHOLD (default: 1.0% allocation),
run 3 targeted web searches:

Search 1: "{ticker} {company} Q1 2026 earnings revenue growth results"
Search 2: "{ticker} valuation analyst price target 2026"
Search 3: "{company} latest news catalyst June 2026"

Use Tavily API (preferred) or Serper API as fallback.
Run searches concurrently using asyncio, rate-limit to max 5 concurrent requests.

For HK-listed positions (e.g. 0700.HK, 9988.HK, 3690.HK):
- Use company name in search, not HK ticker format
- Add "HK" or "Hong Kong" to search query for localisation
- Also search for ADR equivalent if one exists

For ETFs (XLV): search "{ETF_NAME} holdings performance 2026" only.
Skip Search 2 (no analyst price targets for ETFs).

If a search fails: log warning, store empty string, continue.
Do NOT raise exceptions for individual search failures.
```

**Config:**
```python
RESEARCH_THRESHOLD = 1.0      # min allocation % to trigger research
MAX_CONCURRENT_SEARCHES = 5
SEARCH_TIMEOUT_SECONDS = 10
SEARCHES_PER_POSITION = 3
```

### Step 4: Analyse Portfolio (Multi-Pass Claude Agent)

Three sequential Claude API calls.

#### Pass 1 — Per-Position Analysis

```python
SYSTEM_PROMPT_PASS1 = """
You are a senior infrastructure and technology equity analyst
with 15+ years of experience across investment banking and
institutional research. Your mandate covers digital infrastructure,
AI compute, semiconductors, and global technology platforms.

Your analytical framework prioritises:
1. BUSINESS MODEL QUALITY — is revenue real, recurring, and
   from genuine end-customers?
2. EARNINGS TRAJECTORY — recent quarterly results, margin
   direction, FCF generation vs capex burden
3. STRUCTURAL POSITIONING — physical scarcity (power, fab
   capacity), application layer lock-in, or contested model layer
4. VALUATION vs GROWTH — P/E, P/S, EV/EBITDA relative to
   growth rate and sector peers
5. PORTFOLIO CONSTRUCTION — concentration risk, correlation,
   geopolitical exposure (especially China)

CRITICAL RULES:
- Lead with data and evidence. Never assert without citing the search data provided.
- If search data is insufficient, explicitly flag the position.
- Do not hedge excessively. Form a clear view and state it.
- Distinguish between positions where AI enhances the business vs IS the business.
- Note when valuation has already priced in the thesis.

Output format: structured JSON as specified in the user message.
"""

USER_PROMPT_PASS1 = """
Portfolio Summary:
{portfolio_summary_json}

All Positions:
{positions_json}

Research Data Per Position:
{research_data_json}

Analyse each position with allocation >= 1.0%.
For each position return:
{
  "ticker": str,
  "company": str,
  "alloc_pct": float,
  "verdict": "STRONG HOLD" | "HOLD" | "TRIM" | "EXIT" | "ADD",
  "conviction": "HIGH" | "MEDIUM" | "LOW",
  "earnings_quality": str,
  "valuation_assessment": str,
  "structural_position": str,
  "key_risk": str,
  "data_sufficiency": "ADEQUATE" | "THIN",
  "rationale": str
}

Return a JSON array of position analyses. Raw JSON only.
"""
```

#### Pass 2 — Self-Critique

```python
SYSTEM_PROMPT_PASS2 = """
You are a critical research reviewer. Your job is to identify
weaknesses, unsupported assertions, and data gaps in equity research notes.
"""

USER_PROMPT_PASS2 = """
Review the following portfolio analysis for data quality issues.

Analysis to review:
{pass1_output_json}

For each position, identify:
1. Any assertion made without supporting data from the research
2. Any valuation claim not grounded in actual reported multiples
3. Any structural claim that appears assumed rather than evidenced

  },

}
"""
```

#### Pass 3 — Portfolio-Level Synthesis

```python
USER_PROMPT_PASS3 = """
Given the per-position analyses and the critique review,
produce a portfolio-level synthesis covering:

1. CONCENTRATION RISKS
   - Top 5 positions as % of total
   - Correlated exposures (multiple China tech names, multiple AI infra names)
   - Single-factor risks (what breaks the whole portfolio)

2. REALLOCATION RECOMMENDATIONS
   - Specific positions to exit or trim with sizing
   - Specific positions to add to with sizing
   - Target allocation for each change
   - Capital redeployment logic

3. THEMATIC ASSESSMENT
   - AI infrastructure exposure quality (physical scarcity vs model layer vs app layer)
   - China exposure assessment (geopolitical risk vs opportunity)
   - Non-AI anchor positions adequacy

4. TOP 3 ACTIONABLE PRIORITIES

Return JSON:
{
  "concentration_risks": [...],
  "reallocation_recommendations": [
    {
      "action": "EXIT" | "TRIM" | "ADD",
      "ticker": str,
      "current_alloc_pct": float,
      "target_alloc_pct": float,
      "rationale": str,
      "estimated_capital_usd": float
    }
  ],
  "thematic_assessment": {
    "ai_infrastructure_quality": str,
    "china_exposure": str,
    "non_ai_anchors": str
  },
  "top_3_priorities": [str, str, str],
  "overall_portfolio_rating": "STRONG" | "SOUND" | "NEEDS_ATTENTION"
}
"""
```

**Claude API Config:**
```python
MODEL = "claude-sonnet-4-6"
MAX_TOKENS = 8000
```

### Step 5: Format Report

```
Format the analysis JSON into a professional HTML email.
Match the visual style of existing portfolio emails
(Arial font, inline CSS, clean table structure, green/red for positive/negative).

Report sections:
1. Header — date, total portfolio value, day P&L
2. Executive Summary — overall rating + top 3 priorities
3. Reallocation Recommendations — table of actions with current vs target allocation
4. Per-Position Analysis Cards — sorted by allocation %
   - Verdict badge, conviction level, key metrics, rationale, key risk
   - Colour code: green=STRONG HOLD/ADD, amber=HOLD/TRIM, red=EXIT
5. Portfolio Risk Observations — concentration and thematic
6. Footer — generated timestamp, model used

For positions flagged as data_sufficiency=THIN, show a ⚠️ data caveat badge.
```

### Step 6: Deliver Report

```
Email config:
- From: straitsagent@gmail.com
- To: <YOUR_RECIPIENT_EMAIL>
- Subject: "Portfolio Analysis — {date} | {overall_rating} | {top_priority}"
- Body: HTML from Step 5
```

**New PostgreSQL table:**
```sql
CREATE TABLE IF NOT EXISTS portfolio_analysis (
    id                   SERIAL PRIMARY KEY,
    analysis_date        DATE NOT NULL,
    portfolio_value_usd  DECIMAL(12,2),
    day_pnl_usd          DECIMAL(10,2),
    overall_rating       VARCHAR(20),
    position_count       INTEGER,
    analysis_json        JSONB,
    positions_json       JSONB,
    model_used           VARCHAR(50),
    created_at           TIMESTAMP DEFAULT NOW()
);
```

One row per weekly run. Enables week-over-week rating trend tracking.

## 8. Error Handling Standards

| Step | On Failure |
|---|---|
| Step 1 (fetch email) | Fail fast — send alert email |
| Step 2 (parse) | Fail fast — send alert email |
| Step 3 (research) | Degrade — skip failed tickers, continue |
| Step 4 (analyse) | Retry once, then fail fast |
| Step 5 (format) | Fail fast |
| Step 6 (deliver) | Retry 3x with exponential backoff |

## 11. Instructions for Claude Code

When building this workflow:

1. **Start with Step 2 (parser)** — build and test the HTML parser first. Everything downstream depends on clean position data. Or discuss the DB-direct alternative before starting.
2. **Ask for the fixture file** before writing parser code — prompt the owner to paste a sample portfolio email HTML so you can build against real structure.
3. **Build each step as a standalone Python module** testable independently before wiring into Windmill.
4. **Resolve resource naming before writing any code** — `u/admin/portfolio_db` not `pg_main`; confirm Gmail access method.
5. **Do not modify any existing Windmill workflows** — only add new ones.
6. **Add the new schedule to 5.1 Health Check** once the workflow is live.
7. **The system prompts in Section 5, Step 4 are starting points** — iterate on them based on output quality from first test runs.
8. **Decide on F3 overlap** before building — this workflow may replace F3 (Portfolio Review Tool).
